# Dash 보안 및 컴플라이언스 전략 가이드 (Whitepaper)

본 문서는 아동학대정보시스템의 민감 데이터를 다루는 'Dash' 서비스가 어떻게 법적 리스크를 제거하고, 공공기관(보건복지부, 사회보장정보원 등)의 보안 기준을 충족하는지 설명하는 전략 백서입니다.

---

## 1. Dash E2EE(종단간 암호화) 아키텍처

Dash는 **'Zero-Knowledge'** 원칙을 구현하여, 서비스 제공자조차 유저의 데이터를 열람할 수 없는 구조를 채택합니다.

### 1.1. 데이터 흐름 (Data Flow)
1. **암호화 (Mobile App)**: 유저 기기 내에서 공개키 암호화 방식(예: AES-256)을 사용하여 데이터를 암호화합니다. 이때 암호화 키는 서버로 전송되지 않고 기기 내 보안 영역(Secure Element)에만 상주합니다.
2. **전송/보관 (Encrypted Sync)**: 암호화된 바이너리 데이터(`x8fb2...`)만이 Dash 클라우드 서버로 전송되어 보관됩니다. 
3. **복호화 (PC Extension)**: 사무실 PC에서 유저가 로그인하면, 암호화된 데이터를 내려받아 로컬 기기 내에서만 복복화하여 시스템에 주입합니다.

### 1.2. 보안 핵심 포인트
- **서버의 무지(Server Ignorance)**: Dash 서버는 자신이 무엇을 보관하고 있는지 알 수 없습니다. 이는 법적으로 데이터 '처리'의 주체가 아닌 '단순 보관/전송자'로서의 지위를 확보하게 합니다.
- **키 소유권(Key Ownership)**: 복호화 키의 유일한 소유자는 실무자 자신입니다.

## 2. 개인정보보호법(PIPA) 컴플라이언스 대응

공공기관 및 법무 검토를 위한 조항별 대응 논리입니다.

| 관련 조항 | Dash의 준수 및 대응 논리 |
| :--- | :--- |
| **제29조 (안전조치의무)** | 개인정보의 분실·도난·유출을 막기 위해 기술적·관리적 보호조치를 해야 함. Dash는 **최상위 보안 표준인 E2EE**를 적용하여 이를 완벽히 이행함. |
| **제18조 (목적 외 이용/제공)** | Dash는 데이터를 비식별화(암호화)된 상태로 보관하므로, 데이터의 내용을 식별하거나 이용할 수 없음. 따라서 목적 외 이용의 여지가 원천적으로 차단됨. |
| **데이터 처리 위탁 이슈** | Dash는 암호화된 봉투를 전달하는 '디지털 운송업자'에 해당하며, 실제 데이터의 가독(Readability) 권한이 없으므로 민감정보 위탁 절차에서 면복 근거를 가짐. |

## 3. 공공기관 설득을 위한 3대 핵심 논리

### 논리 1: "그림자 업무(Shadow Work)의 양성화"
- 현재 상담원들은 보안 규정 때문에 오히려 개인 메모장, 검증되지 않은 외부 앱, 수기 메모 등을 통해 인권 정보를 관리하고 있음. 이는 관리가 불가능한 보안 사각지대임.
- Dash는 이를 **'암호화된 단일 통로'**로 끌어들여, 데이터의 흐름을 투명하게 하고 유출 리스크를 중앙 관리할 수 있게 함.

### 논리 2: "기술적 거부 능력(Technical Inability)"
- "우리는 데이터를 보여주고 싶어도 보여줄 기술적 능력이 없습니다." 
- 정부의 데이터 제공 요청이나 해킹 사고 시에도, Dash는 암호화된 코드 외에는 줄 수 있는 것이 없음. 이는 기관 입장에서 **'유출 사고 시 책임 소재'**로부터 자유로워질 수 있는 강력한 방어막이 됨.

### 논리 3: "글로벌 보안 트렌드와의 정합성"
- **Apple (Advanced Data Protection)**: iCloud 사진/백업을 E2EE로 전환하여 수사기관의 요청에도 대응 불가하게 설계.
- **의료계 (Patient-Controlled Health Record)**: 환자만 키를 가진 E2EE 기반 의료 기록 관리 모델이 글로벌 표준으로 자리 잡는 추세.

## 4. 보안 사고 시나리오 및 대응 (BGP/DR)

- **시나리오 A: Dash 서버가 해킹당했을 경우**
  - **대응**: 해커가 탈취한 데이터는 암호화된 난수뿐임. 복호화 키는 각 상담원의 개별 기기에 있으므로 실제 아동 정보는 안전하게 보호됨.
- **시나리오 B: 상담원의 폰/PC를 분실했을 경우**
  - **대응**: 기기 자체 보안(생체인증, PIN)과 Dash의 무결성 검증을 통해 2차 보호. 필요 시 원격에서 세션 만료 및 로컬 데이터 삭제 가능.

---

## 5. 결론 및 제언 (Conclusion)

**"Dash는 가장 안전한 방법으로 행정의 효율을 혁신합니다."**

보안과 효율은 반비례한다고 생각하기 쉽지만, Dash는 **종단간 암호화(E2EE)**를 통해 이 모순을 해결했습니다. 사회보장정보원과 보건복지부는 Dash 도입을 통해 실무자의 고충을 해결함과 동시에, 국가 민감 데이터를 이전보다 훨씬 강력한 보안 표준으로 관리할 수 있게 될 것입니다.
