## 야메군 프레임워크 요약

야메군 멘토가 제시한 발표 구조는 다음 6단계다.

```
문제 발견 → 가설 설정 → 정량/정성 리서치 → 인사이트 도출 → 비즈니스 연결 → 한계와 회고
```

### 핵심 인사이트 3가지

**① 비즈니스 연결 = 고객과 서비스 사이의 수저**
> 프로도(고객) → 수저(연결고리) → 밥(서비스)
>
> UX를 개선했다면, "AS-IS의 어떤 문제가 유저가 서비스에 닿지 못하게 막았고,
> TO-BE가 그 막힘을 어떻게 뚫었는가"를 설명하는 것이 비즈니스 연결이다.

**② 사고의 깊이 = 과정의 디테일**
> "배고프다 → 밥 먹었다 → 배불렀다"는 무식한 서술이다.
> 왜 수저를 골랐는지, 왜 이 반찬이 맛있을 거라 가설을 세웠는지,
> 틀렸을 때 무엇을 배웠는지가 지성을 드러낸다.

**③ 성과에 집착하지 말 것**
> 성과가 100% 본인 노력으로 만들어진 것이 아니라면 포장하지 마라.
> 없다면 없는 대로 — 억지 성과보다 정직한 과정이 더 신뢰를 준다.

### STEP 1. 문제 발견

**포트폴리오에서 해야 할 것**: 문제를 "피상적 불편함"이 아닌 "구체적 고통"으로 기술하기.

| 피상적 버전 (NG) | 구체적 버전 (OK) |
|---|---|
| "NCADS 입력이 불편하다" | "상담원은 현장 기록 → 메모장 → 오피스 메신저 → NCADS 정보시스템까지 5단계를 거친다. 한 건 입력에 평균 ___분, 이 중 단순 반복 입력이 ___%를 차지한다." |
| "시스템이 느리다" | "NCADS의 실측 TTFB는 5초 이상. 유저는 탭을 여러 개 띄우지 않는다. 이유: 느린 시스템이 튕길 때 기록이 날아가기 때문." |

### STEP 3. 정량 / 정성 리서치

**포트폴리오에서 해야 할 것**: 데이터가 실제로 의사결정을 바꾼 사례를 보여주기. 수집만 한 데이터는 리서치가 아니다.

**정성 리서치 (이미 보유한 것)**:
- 현장 그림자 관찰 (Shadowing): 상담원 업무 2시간 동행 관찰
- 심층 인터뷰: "왜 이 기능을 안 쓰시나요?" 추적 질문
- 기술 감사: NCADS TTFB 실측

**정량 리서치 (지금 뽑을 수 있는 것)**:

```sql
-- 실제 사용 현황 파악
SELECT 
  COUNT(*) AS total_records,
  SUM(CASE WHEN status = 'Injected' THEN 1 ELSE 0 END) AS ncads_completed,
  SUM(CASE WHEN status = 'Reviewed' THEN 1 ELSE 0 END) AS reviewed,
  SUM(CASE WHEN status = 'Synced' THEN 1 ELSE 0 END) AS synced_only,
  COUNT(reviewer_user_id) AS shared_records,
  COUNT(encrypted_blob) AS e2ee_records
FROM service_drafts;
```

### STEP 4. 인사이트 도출

**포트폴리오에서 해야 할 것**: "리서치 결과 → 그러니까 우리는 X를 해야 한다"의 연결 논리를 명시하기. 인사이트 없는 데이터는 나열일 뿐이다.

**Dash에서 도출된 인사이트 목록**:

| 리서치 발견 | 도출된 인사이트 | 실제 한 액션 |
|---|---|---|
| 유저가 현장에서 메모 후 나중에 입력 | 현장→사무실 이동 구간이 공백이다 | 모바일 초안 작성 기능 |
| 상담원이 "대리님 컨펌" 단계를 거친다 | 공유/검토 플로우가 필요하다 | 리뷰어 공유 링크 기능 |
| NCADS 기입이 반복 클릭 위주 | 자동화 가능한 구조 | Chrome 확장프로그램 자동 주입 |
| 민감한 아동 정보가 포함됨 | 암호화 없이는 공유가 위험 | E2EE + Vault 설계 |

---

### STEP 5. 비즈니스 연결

**포트폴리오에서 해야 할 것**: Dash라는 수저가 없으면 상담원(프로도)이 NCADS(밥)에 어떻게 닿는지, Dash가 그 경로를 어떻게 바꿨는지 서술.

**야메군 공식 적용**:

```
[AS-IS] 상담원 → 클로바노트 → 개인 메모장 → 오피스 메신저 컨펌 → NCADS 수기 입력
                                                        ↑ 여기서 정보 유실, 중복 입력, 시간 낭비 발생

[TO-BE] 상담원 → Dash 모바일 앱 → (리뷰어 승인) → Chrome 확장 1클릭 → NCADS 자동 주입
                      ↑ Dash = 수저 (연결고리)
```

**비즈니스 연결의 구체적 표현 (발표용)**:

> "저는 UX를 개선한 것이 아닙니다. 상담원이 NCADS에 닿기까지의 경로 자체를 5단계에서 2단계로 줄였습니다. 이 과정에서 Dash는 단순한 입력 보조 앱이 아니라, 현장 기록과 공식 정보시스템 사이의 유일한 연결고리가 되었습니다."

**지금 확인해야 할 팩트 (TODO)**:

- [ ] TO-BE 2단계 각각의 소요 시간 실측
- [ ] `status = 'Injected'` 건수 → "실제 NCADS까지 연결된 사례" 수치화

---

### STEP 6. 한계와 회고

**포트폴리오에서 해야 할 것**: 성과를 억지로 만들지 말고, 정직한 한계를 전략적으로 표현하기. 한계를 아는 사람이 다음 단계를 설계할 수 있다.

**현재 Dash의 솔직한 한계 목록**:

| 한계 | 이유 | 다음 단계 |
|---|---|---|
| 실사용자 수가 소규모 | 조직 내 비공식 도구로 시작됨 | 기관 단위 도입 제안 필요 |
| 성과 수치(시간 절감)를 정확히 측정 못 함 | 사전 측정 설계 안 됨 | 향후 A/B 측정 방법론 필요 |
| NCADS 자동 주입이 시스템 업데이트에 취약 | DOM 기반 selector 의존 | selector 안정화 레이어 필요 |
| iOS 앱스토어 미출시 | 개인 계정 심사 중 | 기관 계정 or TestFlight 배포 |

**회고 표현 방식 (발표용)**:

> "이 프로젝트에서 제가 가장 크게 배운 것은, 측정 설계를 처음부터 했어야 했다는 점입니다. 효과가 있다는 확신은 있지만, 숫자로 증명할 수 없는 것들이 있습니다. 다음 프로젝트에서는 프로덕트를 만들기 전에 측정 방법을 먼저 설계하겠습니다."

## 발표 스크립트 구조 (5분 기준)

```
[0:00 - 0:40] 문제 발견
  "상담원들은 한 건의 DB를 입력하기 위해 5개의 도구를 오갑니다. 
   이 구조를 발견하게 된 계기는..."

[0:40 - 1:20] 가설 설정 + 실패
  "처음엔 UI 문제라고 생각했습니다. 그래서 다중 창 제어 UI를 만들었는데, 
   아무도 쓰지 않았습니다. 이유를 추적해보니..."

[1:20 - 2:20] 리서치 + 인사이트
  "현장에 2시간 동행하며 알게 된 것은... 
   이 발견이 제품 방향을 완전히 바꿨습니다."

[2:20 - 3:30] 비즈니스 연결 (핵심)
  "Dash는 현장 기록과 공식 정보시스템을 연결하는 수저입니다.
   AS-IS 5단계를 TO-BE 2단계로. 구체적으로는..."

[3:30 - 4:20] 실제 구현 결과
  "현재 _건의 기록이 Dash를 통해 NCADS까지 연결됐습니다.
   이 중 리뷰어 공유 기능을 통한 건은 ___건입니다."

[4:20 - 5:00] 한계와 회고
  "가장 아쉬운 건 측정 설계를 초반에 하지 않은 것입니다.
   다음에는..."
```
