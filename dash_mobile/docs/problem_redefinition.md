# Dash — 문제 정의서 (Problem Definition)
**버전**: v2.0  
**최종 수정**: 2026-05-07  
**문서 목적**: Dash가 해결하는 상담원의 핵심 문제를 계층적으로 정의하고, 솔루션 전략과의 연결 고리를 명확히 함

---

## 1. 한 줄 문제 정의

> **"아동보호 상담원은 '상담'이 아닌 '기록 정리'에 가장 많은 에너지를 쏟고 있다."**

---

## 2. 유저 컨텍스트: 우리의 유저는 누구인가

| 항목 | 내용 |
|:---|:---|
| **직군** | 전국 아동보호전문기관 소속 상담원 (신입 간사 ~ 중간관리자) |
| **핵심 임무** | 피해아동 현장 방문 → 상담 기록 → 국가 DB 시스템 기입 |
| **주요 도구** | 국가아동학대정보시스템 (레거시 웹 시스템), 클로바노트, 개인 메모장, 오피스 메신저 |
| **규제 환경** | 개인정보보호법 및 아동복지법에 따른 엄격한 정보 관리 의무 |
| **업무 특성** | 현장(외근) ↔ 사무실(기입) 간 물리적 분리, 도제식 컨펌 문화 |

---

## 3. 발견된 문제들 (레이어별 분류)

### Layer 1 — 표면적 문제 (Surface Pain): 유저가 호소하는 것

> *"DB 입력하는 프로그램 좀 만들어줘."*

- 입력 UI가 복잡하고 필드가 많아 클릭 수가 많다
- 여러 창을 오가며 복사-붙여넣기를 반복해야 한다
- 입력할 항목이 규격화되어 있지 않아 헷갈린다

---

### Layer 2 — 구조적 문제 (Structural Pain): 실제로 관찰된 것

유저의 하루는 다음 **파편화된 5개 도구의 사슬**로 이루어진다.

```
[현장] 클로바노트·수기 메모
    ↓ (이동 중)
[이동] 개인 메모장에 내용 정리·정제
    ↓ (복사→붙여넣기)
[협업] 오피스 메신저로 상급자(대리님)에게 초안 전송
    ↓ (수정 사항 재수신 후 복사)
[반영] 수정된 내용을 다시 메모장에 붙여넣고 최종본 확정
    ↓ (복사→붙여넣기)
[기입] 느리고 불안정한 국가 시스템에 타이핑
```

**핵심 비용**: 동일한 데이터를 최소 **4~5회 재가공·복사**하며 인지 에너지 소진

---

### Layer 3 — 심층 문제 (Root Cause): 가장 치명적인 공포

> *"사실 제일 짜증나는 건 '시스템 느린 거' + '튕길 수도 있는 거'에요. 여러 창 안 열어놓는 이유도 탭 많으면 브라우저가 무거워져서 튕길까봐 그런 거거든요."*  
> — 유저 인터뷰, 2026.03.12

**진짜 문제**: 국가 시스템은 레거시 인프라 기반으로 TTFB가 3~5초 이상이며, 세션 만료로 인한 **자동 로그아웃이 언제든 발생할 수 있다.**  
→ 시스템에 오래 머물수록 수천 자의 기록이 통째로 사라질 위험이 높아진다.  
→ 이 **'데이터 증발의 공포'**가 상담원의 모든 업무 행동 방식을 결정한다.

---

## 4. 문제의 심각도 분석

| 문제 유형 | 발생 빈도 | 발생 시 피해 | 우선순위 |
|:---|:---|:---|:---|
| 입력 UI 복잡성 (클릭 수 多) | 높음 | 낮음 (불편함) | ★★☆ |
| 파편화된 도구 간 반복 복사 | 매일 발생 | 중간 (인지 에너지 소진, 시간 손실) | ★★★ |
| 상급자 컨펌 지연 | 자주 발생 | 중간 (전체 파이프라인 블로킹) | ★★★ |
| **시스템 튕김에 의한 데이터 유실** | 낮음 | **치명적** (업무 전체 재시작) | **★★★★★** |

> **PM 판단**: 빈도보다 **심각도(Severity)가 높은 문제**를 최우선 해결 대상으로 설정. 단 한 번의 데이터 유실은 30분~1시간의 업무 Rework를 유발한다.

## 5. 최종 문제 정의 (재정립)

### ❌ AS-IS 가설 (틀린 가설)
> "국가 시스템의 입력 UI가 파편화되어 있어 클릭 수가 많고 업무 시간이 길어진다."

→ **초점**: 기술적 효율성 (UI 개선)

---

### ✅ TO-BE 정의 (검증된 정의)
> **"상담원은 불안정한 국가 시스템에 대한 공포와, 파편화된 도구들 사이의 반복적 데이터 재가공이라는 이중 고통으로 인해, 정작 가장 중요한 '상담'에 집중하지 못하고 있다."**

→ **초점 1**: 심리적 안전감 — 데이터가 절대 사라지지 않는다는 확신  
→ **초점 2**: 워크플로우 통합 — 도구 전환 없이 현장에서 DB까지 하나의 흐름  
→ **초점 3**: 체류 시간 최소화 — 불안정한 시스템에 머무는 시간을 극단적으로 줄임

---

## 6. Dash가 해결하는 3가지 핵심 문제

### Problem 1. 데이터 증발의 공포 (Data Loss Anxiety)
**상황**: 상담원은 국가 시스템의 불안정성 때문에 수기 메모와 메모장을 병행하는 '방어적 업무 패턴'을 취하고 있다.  
**Dash 해결**: 시스템 외부(Dash 앱)에서 모든 작업을 완료한 뒤, 시스템에는 단 1초 만에 '주입(Inject)'만 한다. 시스템 체류 시간 **90% 단축**.

---

### Problem 2. 도구 파편화에 의한 인지 에너지 소진 (Cognitive Overload)
**상황**: 동일한 내용을 클로바노트 → 메모장 → 메신저 → 시스템까지 4~5회 재가공하며 퇴근 전 에너지가 바닥난 상태로 DB 입력을 마주한다.  
**Dash 해결**: 현장에서 앱으로 입력한 데이터가 PC 확장 프로그램으로 자동 동기화되는 **단일 파이프라인**을 구축. 복사-붙여넣기 횟수 **75% 감소**.

---

### Problem 3. 도제식 컨펌 프로세스의 비효율 (Review Loop Friction)
**상황**: 신입 간사가 작성한 초안을 대리님이 검토하는 과정에서 '메신저 전송 → 복사 수정 → 재전송 → 재복사'의 4회 왕복이 발생하며, 이 과정이 DB 입력 지연의 50% 이상을 차지한다.  
**Dash 해결**: **Shared Draft Link**를 통해 대리님이 Dash 웹에서 직접 수정하면 간사의 앱과 확장 프로그램에 즉시 최종본이 동기화된다. 컨펌 사이클 **1회로 단축**.

---

## 7. 문제-솔루션 매핑 (Problem-Solution Fit)

```
[현장] 수기 메모의 분실 위험
    ↓ Dash Mobile App → 현장에서 즉시 구조화된 DB 형식으로 기록

[이동] 반복적인 데이터 정제 부담
    ↓ AI 자동 정제 → 구어체 메모를 DB용 소견으로 자동 변환

[협업] 메신저 컨펌 프로세스의 비효율
    ↓ Shared Draft Link → 공동 편집 후 즉시 동기화

[기입] 국가 시스템 체류 리스크
    ↓ Chrome Extension → Magazine Injection (1초 내 전 필드 자동 입력)
```

## 8. 가설 검증 지표

| 지표 | 측정 방법 | 목표 |
|:---|:---|:---|
| **시스템 체류 시간** | 이전 평균(~10분) vs Dash 사용 후 | **1분 이하** |
| **사용 도구 수** | 상담 완료~DB 저장까지 거치는 앱·도구 개수 | **4~5개 → 1개** |
| **복사-붙여넣기 횟수** | 직접 관찰 또는 유저 보고 | **75% 감소** |
| **컨펌 사이클** | 메신저 왕복 횟수 | **4회 → 1회** |
| **심리적 안전감** | "데이터 유실 걱정 없다" 응답률 | **80% 이상** |
| **업무 전진 배치율** | 사무실 복귀 전 완료된 기록의 비율 | **50% 이상** |
