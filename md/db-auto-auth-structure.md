# DB Auto 유저 인증 구조 설계

## 1. 개요
* **서비스 형태**: 크롬 확장프로그램
* **타겟 고객**: 굿네이버스 아동보호전문기관 상담원
* **도메인 특성**: 전국 굿네이버스 법인 상담원 이메일 도메인이 모두 `@gnk.or.kr`로 동일하기 때문에, 이메일 도메인만으로는 개별 기관 단위를 구분할 수 없습니다.

---

## 2. 인증 전략 (2단계 도입)

### 1단계: 이메일 화이트리스트 방식 (개별 인증)
* 서비스 도입 초기, 실제 계약을 맺은 기관 소속 상담원의 **이메일을 Supabase에 직접 등록(화이트리스트)**하여 개별적으로 인증 및 접근을 허용합니다.

### 2단계: 도메인 전체 허용 방식 (엔터프라이즈 인증)
* 추후 굿네이버스 본부와 Enterprise 계약이 성사될 경우, 개별 화이트리스트 등록 없이 **`@gnk.or.kr` 도메인을 가진 모든 사용자의 접근을 허용**하는 방식입니다.

---

## 3. 데이터베이스(Supabase) 구조 설계
처음부터 1단계와 2단계를 모두 수용할 수 있는 유연한 구조로 설계합니다. 이를 통해 1단계에서 2단계로 전환할 때 **별도의 데이터 마이그레이션 없이** 데이터 값과 인증 로직 수정만으로 원활한 전환이 가능합니다.

### 테이블 스키마

#### `organizations` (기관 정보)
| 컬럼명 | 타입 | 설명 | 비고 |
| --- | --- | --- | --- |
| `id` | uuid (PK) | 기관 고유 ID | |
| `name` | text | 기관명 | |
| `contract_type` | text | 계약 형태 | `individual` (1단계) 또는 `enterprise` (2단계) |
| `domain` | text | 이메일 도메인 | 기본값: `gnk.or.kr` |
| `active` | boolean | 기관 활성화 여부 | |

#### `users` (유저 정보)
| 컬럼명 | 타입 | 설명 | 비고 |
| --- | --- | --- | --- |
| `id` | uuid (PK) | 회원 고유 ID | |
| `email` | text | 회원 이메일 | |
| `org_id` | uuid (FK) | 소속 기관 ID | `organizations` 테이블의 `id` 참조 |
| `active` | boolean | 계정 활성화 여부 | |

---

## 4. 확장성 (1단계 ➔ 2단계 전환 방식)
2단계(본사 엔터프라이즈 계약)로 전환 스펙이 발생하면, 다음 두 가지만 반영하여 빠르게 전환할 수 있습니다.
1. `organizations` 테이블에서 해당 기관의 `contract_type` 컬럼 값을 `individual`에서 **`enterprise`**로 변경
2. 인증 서버(또는 Supabase Auth 로직)에서 `contract_type`이 `enterprise`인 경우, `domain`(`gnk.or.kr`)이 일치하면 화이트리스트 테이블 검증 없이 가입 및 로그인을 허용하도록 **로직 수정**
