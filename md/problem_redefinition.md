# Dash 프로젝트: 문제 정의 재정립 (Problem Redefinition)

본 문서는 2026년 3월 12일 진행된 유저 인터뷰(안서영 간사)를 통해 도출된 인사이트를 바탕으로, 서비스의 핵심 가치와 문제 해결 방향을 전면 재정의한 기록입니다.

## 1. 프로젝트 발단: "요청과 실재 사이의 간극"

- **Literal Request**: "윤수쌤, DB 입력하는 프로그램 좀 만들어줘."
- **기획자의 첫 가설**: "유저가 입력하는 과정이 귀찮고 복잡해서 시간을 단축하고 싶어하는구나." (UI 효율성 집중)
- **인터뷰를 통한 새 발견**: 실제 유저는 단순히 '귀찮음'을 넘어, 시스템의 불안정성(튕김)에 의한 **'데이터 증발의 공포'**를 가장 큰 페인 포인트로 느끼고 있었음.
- **결론**: 유저의 요청(DB 입력 프로그램)은 유효했으나, 그 목적은 단순 효율이 아닌 **'데이터 안전 확보 및 시스템 체류 시간 최소화'**에 있었다는 본질적 가설로 진화.

---

## 2. 기존 문제 정의 vs 재정립된 문제 정의

### [AS-IS] 기존의 가설 (Desk Research 기반)
- **문제**: "국가 시스템의 입력 UI가 파편화되어 있고 클릭이 많아 업무 시간이 길어진다."
- **솔루션**: "다중 창 제어 UI(Carousel)와 고밀도 버튼 그리드를 통해 입력 동선을 최적화한다."
- **초점**: **기술적 효율 성(Efficiency)**

### [TO-BE] 재정립된 문제 (User Interview 기반)
- **문제**: "국가 시스템은 극도로 느리고 불안정하여, **기입 중 튕김(Data Loss)에 대한 공포**가 실무자의 가장 큰 페인 포인트다."
- **솔루션**: "시스템 외부에서 데이터를 완벽히 준비(Draft)하고, 시스템 내에서는 최소한의 시간 동안 주입(Inject)만 하고 빠져나온다."
- **초점**: **심리적 안정감 및 체류 시간 최소화 (Safety & Speed)**

---

## 3. 유저 인터뷰 및 다각도 검증 (Discovery & Validation)

단순히 한 명의 인터뷰이(N=1)의 의견에 의존하지 않고, 다음과 같은 **'교차 검토(Triangulation)'** 과정을 통해 가설의 정당성을 확보했습니다.

### 3.1. 기술적 실태 조사 (Technical Audit)
- **현상**: 국가아동학대정보시스템은 레거시 인프라 기반으로, 피크 타임 시 응답 지연(TTFB)이 3~5초 이상 발생.
- **리스크**: 일정 시간 이상 입력 폼이 열려 있을 경우 세션 만료로 인한 자동 로그아웃 위험이 존재함.
- **결론**: **"시스템 체류 시간 자체가 리빙 리스크(Living Risk)"**라는 기술적 판단을 내림.

### 3.2. 행동 패턴 관찰 (Cross-Observation)
- **추가 사례**: 인터뷰이 외 2명의 추가 실무자 업무 관찰.
- **발견된 공통 행동**: 3명 모두 듀얼 모니터를 사용함에도 브라우저 탭은 단 1개만 열어두고 있었음. 
- **공통 답변**: "탭을 여러 개 띄우면 브라우저가 무거워지고, 그러다 튕기면 방금 기록한 수천 자의 상담 기록이 날아간다."
- **인사이트**: 다중 창 제어에 대한 거부감은 개인의 취향이 아닌, '데이터 손실 방지'를 위한 **집단적 방어 기제**임이 확인됨.

### 3.3. 문제의 임팩트 분석 (Severity vs Frequency)
- **클릭 최적화(기존 가설)**: 빈도는 높으나 해결하지 않아도 업무 수행에는 지장이 없음.
- **체류 시간 최소화(신규 가설)**: 발생 빈도는 낮을 수 있으나 한번 발생 시 '업무 재시작(Rework)'이라는 치명적인 비용이 발생함.
- **PM적 판단**: 빈도 개선보다는 **'치명적 페인 포인트(Critical Pain Point)' 해결**에 우선순위를 두기로 결정.

## 4. 유저 인터뷰 핵심 인사이트 (3/12)

> *"사실 제일 짜증나는 건 ‘시스템 느린 거’ + ‘튕길 수도 있는 거’에요. 여러 창 안 열어놓는 이유도 탭 많으면 브라우저가 무거워져서 튕길까봐 그런 거거든요."*

1.  **방어적 업무 패턴**: 유저는 시스템의 불안정성 때문에 스스로 '멀티태스킹'을 포기하고 '순차 처리'를 선택하고 있음. (기존 Dash의 다중 창 제어 가설이 현장 제약에 가로막힘)
2.  **전환 비용의 고통**: 외부 메모(엑셀, 수기)를 보며 느린 시스템 창에 하나씩 기입하는 과정에서 발생하는 시간 지체 자체가 리스크로 작용함.
3.  **대기열(Queue)에 대한 니즈**: 상담원 본인이 제안한 *"미리 써놓고, 창 열면 하나 골라서 뙇 집어넣기"*는 시스템 체류 시간을 극단적으로 줄이려는 생존 본능적 해결책임.

---

## 4. 전략적 피봇 방향: 'Magazine Fill' 전략

Dash는 이제 '입력 보조 도구'를 넘어, 불안정한 시스템으로부터 상담원을 보호하는 **'데이터 세이프 존(Safe Zone)'**이 됩니다.

- **Phase 1. Draft (시스템 밖)**: 시스템에 로그인하지 않은 상태에서도Dash 앱에서 상담 내용을 신속하게 작성 및 임시 저장.
- **Phase 2. Magazine (탄창화)**: 작성된 여러 건의 사례를 Dash 리스트에 '탄알'처럼 적재.
- **Phase 3. Inject (주입)**: 시스템 창을 열자마자 리스트에서 해당 아동을 선택하여 **1초 내외로 모든 필드를 채우고 즉시 저장 버튼 클릭.**

## 5. 최종 진화: UI 개선을 넘어 '워크플로우 파이프라인'으로 (Process Innovation)

3월 13일 추가 인터뷰를 통해, 단순히 "입력이 힘들다"가 아닌 **"입력에 도달하기까지의 과정이 파편화되어 있다"**는 더 거대한 문제를 발견했습니다.

### 5.1. AS-IS: 파편화된 도구의 사슬 (Disconnected Chain)
현재 상담원들은 하나의 사례를 입력하기 위해 다음 도구들을 전전합니다.
1. **현장**: 클로바노트(녹음) or 수기 메모
2. **이동**: 메모장에 내용 정리 및 정제
3. **협업**: 오피스 메신저로 상급자(대리님)에게 검토 요청 및 수정 피드백 수신
4. **최종**: 메신저 내용을 복사하여 불안정한 국가 시스템에 기입

=> **문제**: 이 과정에서 '데이터의 휘발'과 '반복적인 복사-붙여넣기'로 인해 인지적 에너지가 바닥난 상태로 마지막 DB 입력을 마주하게 됩니다.

### 5.2. Dash TO-BE: 엔드투엔드 데이터 파이프라인 (Unified Pipeline)
Dash는 이제 단순한 입력기가 아닌, **상담의 시작부터 DB 저장까지를 하나로 잇는 허브**가 됩니다.

- **[현장] 사전 셋팅**: Dash 모바일 앱에서 DB 양식에 맞춰 미리 항목을 선택하고 상담 내용을 기록.
- **[이동] AI 자동 정제**: Dash 내장 AI가 어수선한 상담 메모를 DB 입력용 소견으로 자동 요약/정리.
- **[협업] 원클릭 공유**: 정제된 내용을 바로 오피스 메신저로 공유하여 컨펌 프로세스 단축.
- **[최종] 지능형 주입**: 사무실 복귀 후, Dash 확장 프로그램에 이미 동기화된 리스트를 선택해 시스템에 즉시 주입.

---

## 6. 가설 검증 지표 (최종 정교화)

1.  **과정 단순화 지표**: 상담 완료 후 DB 저장까지 사용되는 도구(App/Web)의 개수 변화 (기존 4~5개 -> Dash 1개).
2.  **업무 전진 배치율**: 사무실 책상 앞이 아닌, 현장/이동 중에 완료되는 업무의 비율.
3.  **데이터 무결성**: 메신저-메모장-DB를 오가는 과정에서 발생하는 오기입 발생률 0%.
