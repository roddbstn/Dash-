# Dash: 사용자 사용성 테스트(UT) 및 검증 시나리오 기획안

이 문서는 'Dash' 서비스의 본질적인 가치를 검증하고, 사회복지사의 실무 환경에서의 페인 포인트를 해결하는지 확인하기 위한 UT 시나리오입니다. 단순한 기능 점검을 넘어 사용자의 의도와 감정, 그리고 데이터에 기반한 성과 지표를 도출하는 데 집중합니다.

---

## 1. 테스트 배경 및 목적 (The "Why")
- **기획 의도**: 상담원이 여러 개의 DB 창을 열어놓고 반복 입력하는 과정에서 발생하는 '인지적 과부하'와 '물리적 시간 소모'를 Dash가 얼마나 효과적으로 해결하는가?
- **핵심 목표**: Dash의 핵심 가치인 '다중 창 관리 및 자동 입력'이 사용자에게 실제 'WOW' 포인트(감동의 지점)를 제공하는지 검증.

---

## 2. 사전 유저 인터뷰 기획: 현장의 '진짜 문제' 정의하기
유저 스토리(User Story)를 확정하기 전, 실제 사회복지사의 업무 컨텍스트를 깊이 있게 파악하여 가설의 현실성을 확보합니다.

### 2.1. 인터뷰 목적 (The Goal)
- 단순한 '번거로움'을 넘어, 실무자가 DB 입력 과정에서 느끼는 **심리적 압박감, 물리적 제약, 인지적 오류의 구체적 지점**을 발견.
- 기획자의 가설("복사-붙여넣기가 불안하다")이 실제 현장에서도 유효한지, 아니면 더 본질적인 '왜(Why)'가 존재하는지 확인.

### 2.2. 인터뷰 대상 및 방식
- **대상**: 아동보호전문기관/복지관 등에서 1년 이상 DB 입력 업무를 수행해온 실무자 3~5명.
- **방식**: 
    - **관찰 조사(Shadowing)**: 실제 DB 입력 업무를 하는 모습을 옆에서 지켜보며, 어떤 시점에 한숨을 쉬거나 마우스를 헤매는지 관찰.
    - **심층 인터뷰(FGI/IDI)**: 관찰에서 발견된 특정 행동에 대해 "왜 그렇게 하셨나요?"라고 질문.

### 2.3. 핵심 인터뷰 질문 리스트
- **업무 컨텍스트**: "DB 입력을 위해 보통 몇 개의 창을 띄워놓으시나요? 그때 머릿속은 어떤 상태인가요?"
- **프로세스 디테일**: "정보를 옮길 때 본인만의 노하우나 규칙이 있나요? (예: 메모장에 먼저 적는다 등) 왜 그렇게 하시나요?"
- **감정 및 통증 수치**: "하루에 5명 이상의 데이터를 연속으로 넣어야 할 때, 가장 회의감이 드는 순간은 언제인가요?"
- **의도 파악**: "시스템이 자동으로 넣어준다고 할 때, 가장 먼저 드는 생각은 '우와 편리하다'인가요, 아니면 '믿을 수 있을까'인가요?"

### 2.4. 결과 활용 (Output)
- 인터뷰 내용을 바탕으로 **기존(AS-IS) 고객 여정 지도**를 작성.
- 본 문서의 '3. 유저 스토리'를 실무자가 직접 한 말(Quote) 중심으로 현실성 있게 업데이트.

## 3. 가설 설정 (Hypothesis X-Y-Z)
가설을 통해 기획의 논리적 근거를 마련합니다.
- **문제(X)**: 3명 이상의 아동 정보를 입력할 때, 브라우저 탭을 일일이 클릭하며 대상을 확인하는 과정에서 혼란이 생기고 입력 시간이 기하급수적으로 늘어남.
- **해결(Y)**: Dash의 '다중 창 캐러셀 인터페이스'와 '전체 창 일괄 입력' 솔루션 제공.
- **성과(Z)**: 기존 수동 입력 방식 대비 **업무 소요 시간 50% 이상 단축** 및 **입력 대상 착오 발생률 0%** 달성.

---

## 4. 유저 스토리 (User Story)
사전 인터뷰를 통해 정교화될 **예상 페르소나의 목소리**입니다.
> "나는 아동보호전문기관의 상담원으로서, 오늘 방문 상담을 마친 3명의 피해 아동 정보를 시스템에 입력해야 한다. 단순히 복붙이 귀찮은 게 아니라, **똑같은 정보를 여러 번 반복해서 넣고 있는 나 자신이 소모적으로 느껴질 때가 많다.** Dash를 통해 이런 기계적인 일에서 벗어나 더 중요한 사례 관리에 집중하고 싶다."

---

## 5. 테스트 시나리오 및 과업 (User Tasks)

### [Task 1] 다중 창 인지 및 수동 입력 전환
- **상황**: PC에 3명의 아동(강*정, 이*수, 박*규) 창이 열려 있습니다.
- **과업**: Dash의 수동 입력 탭에서 3명의 아동을 캐러셀로 스크롤하며 전화를 통한 상담 내역을 각각 입력해보세요.
- **관찰 포인트**: 
    - 상단 고정 탭 바를 통해 현재 누구의 창을 편집 중인지 직관적으로 이해하는가?
    - 캐러셀 스크롤 시의 반응이 자연스러운가?

### [Task 2] 엑셀 업로드를 통한 일괄 자동화 경험
- **상황**: 외근 중 정리한 엑셀 파일에 3명의 데이터가 들어있습니다.
- **과업**: 엑셀 파일을 업로드하고, Dash가 추출한 데이터가 맞는지 확인한 후 '전체 창에 DB 입력' 버튼을 눌러보세요.
- **관찰 포인트**: 
    - 추출된 데이터가 실제 DB 창의 정보와 일치한다는 것을 확인할 때 사용자의 표정 변화(안심/WOW).
    - 자동 입력 완료 후의 심리적 해방감 정도.

---

## 6. 심층 인터뷰 가이드 (Focus on "Why" & Emotions)
단순한 "좋아요?"가 아닌, 사용자의 의도와 '눈치'를 파악하는 질문들입니다.

### 단계별 감정 분석
- "창을 전환하는 과정에서 어느 순간에 가장 짜증이 나거나 불안하셨나요? 왜 그렇게 느끼셨죠?"
- "자동 입력 버튼을 누르기 직전에 망설임이 있었다면, 무엇 때문이었나요? (시스템에 대한 불신, UI의 불명확성 등)"

### 가설 검증 질문
- "만약 이 기능이 실무에 도입된다면, 귀하의 퇴근 시간이 얼마나 빨라질 것으로 예상하시나요? 그 근거는 무엇인가요?"
- "이 서비스에서 가장 '기분이 좋았던' 순간은 언제인가요? 왜 그 부분이 핵심이라고 생각하시나요?"

## 7. 성과 지표 및 산출물 (KPIs & Deliverables)

### 정량적 지표 (Performance Indicators)
- **Task completion time**: 기존 방식 vs Dash 방식 시간 비교.
- **Error rate**: 오입력 또는 미입력 발생 횟수.
- **Click count**: 입력 완료까지 필요한 클릭 수의 변화.

### 정성적 산출물 (Customer Journey Map)
테스트 후 아래 단계별로 사용자의 감정과 페인 포인트를 정리하여 **고객 여정 지도**를 완성합니다.
1. **인지**: Dash 실행 및 다중 창 연결 확인.
2. **탐색**: 엑셀 업로드 또는 수동 입력 데이터 확인.
3. **실행**: 자동 입력 버튼 클릭.
4. **검증**: 시스템에 실제 반영된 내용 확인.
5. **종료**: 전체 입력 완료 처리.

---

## 8. 결론 및 향후 계획 (Intent Analysis)
- UT 결과를 통해 서비스의 **레거시(Legacy)**를 남깁니다.
- "우리가 세운 가설이 얼마나 적중했는가?"
- "사용자가 정말로 WOW를 느꼈는가, 아니면 기능적 편리함에 그쳤는가?"
- 본 테스트의 결과는 차기 디자인 시스템 고도화 및 데이터 구조 설계의 핵심 근거가 됩니다.
